# VPN Server 部署过程

## 2026-04-23

### 需求

在 Ubuntu 22.04 x86_64 (公网 IP `74.48.78.17`) 上搭建 VPN 服务器，要求：
1. 能在 Clash 上订阅
2. 支持通过 URL 分发配置

### 决策

1. **方案选择**：Trojan 协议（更稳定）
   - 原因：用户验证可用，连接稳定

2. **部署方式**：Docker 容器化（TeddySun 镜像）
   - 避免本机裸部署，便于迁移

3. **域名**：使用 nip.io 免费域名
   - Let's Encrypt 不支持裸 IP 申请证书
   - nip.io 提供 `74-48-78-17.nip.io` 格式的免费泛解析

4. **分发方式**：独立 nginx 容器提供 HTTP 下载服务（8080 端口）

### 过程

#### 第一阶段：部署 Trojan 协议

1. 拉取 `teddysun/trojan` Docker 镜像
2. 生成密码（UUID）：`12a90b0b-0465-429f-8f36-14b2ebdaef5d`
3. 配置 Trojan 服务端（config.json）
4. 启动 Trojan 容器（443 端口）
5. 创建 Docker 网络 `trojan-net`

**Trojan 服务端配置**：
```json
{
    "run_type": "server",
    "local_addr": "0.0.0.0",
    "local_port": 443,
    "remote_addr": "127.0.0.1",
    "remote_port": 80,
    "password": ["12a90b0b-0465-429f-8f36-14b2ebdaef5d"],
    "log_level": 1,
    "ssl": {
        "cert": "/etc/trojan/certs/fullchain.pem",
        "key": "/etc/trojan/certs/privkey.pem",
        "alpn": ["http/1.1"],
        "reuse_session": true,
        "session_ticket": false
    }
}
```

#### 第二阶段：配置下载服务

1. 启动 `nginx:alpine` 容器（8080 端口）
2. 配置 nginx 提供 `/clash.yaml` 下载

**nginx 配置**：
```nginx
server {
    listen 8080;
    server_name _;
    location /clash.yaml {
        alias /usr/share/nginx/html/clash.yaml;
        default_type text/plain;
    }
    location / {
        return 404;
    }
}
```

#### 第三阶段：客户端配置

Clash 代理配置（原生 Trojan，非 WebSocket）：
```yaml
proxies:
  - name: "VPN-Trojan"
    type: trojan
    server: 74.48.78.17
    port: 443
    password: "12a90b0b-0465-429f-8f36-14b2ebdaef5d"
    skip-cert-verify: true
    sni: 74-48-78-17.nip.io
```

### 调试记录

#### 2026-04-23 14:00 - WebSocket 配置错误

**问题**：Clash 客户端连接超时

**原因**：Clash 配置包含 `network: ws` 和 `ws-opts` 字段，但 TeddySun trojan 镜像是原生 Trojan 协议，**不支持 WebSocket**。

**错误日志**（服务端）：
```
[ERROR] 74.48.78.17:48462 SSL handshake failed: stream truncated
```

**解决**：移除 WebSocket 相关配置，使用原生 Trojan 直连模式。

#### 2026-04-23 14:20 - 当前状态

**症状**：客户端连接仍然超时

**服务端验证**：
- TCP 443 端口：`nc -zv 74.48.78.17 443` ✅ 连接成功
- 下载服务：`curl http://74.48.78.17:8080/clash.yaml` ✅ 返回 1269 字节有效 YAML
- TLS 证书：✅ 有效，CN=74-48-78-17.nip.io
- Trojan 容器运行中 ✅

**服务端 Trojan 日志（错误类型）**：
```
[ERROR] 66.132.172.182:45898 SSL handshake failed: version too low
[ERROR] 59.72.97.73:60513 SSL handshake failed: Connection reset by peer
[ERROR] 74.48.78.17:48462 SSL handshake failed: stream truncated
[WARN] 74.48.78.17:54838 not trojan request, connecting to 127.0.0.1:80
```

**分析**：
- `version too low` - 客户端 TLS 版本不支持
- `Connection reset by peer` - 连接被重置，可能是协议不匹配
- `stream truncated` - 数据不完整，客户端发送的数据不符合 Trojan 协议格式
- `not trojan request` - 客户端发送的不是 Trojan 协议数据

### 服务端验证结果

| 测试 | 结果 |
|------|------|
| 下载服务 | ✅ `curl http://74.48.78.17:8080/clash.yaml` 返回 200，1269 字节 |
| Trojan 容器 | ✅ 正常运行，日志显示 `trojan service (server) started at 0.0.0.0:443` |
| TCP 443 端口 | ✅ `nc -zv 74.48.78.17 443` 连接成功 |
| TLS 握手 | ✅ 证书有效，CN=74-48-78-17.nip.io |
| 客户端连接 | ❌ 超时，Trojan 日志显示多种握手错误 |

### 经验总结

1. **nip.io 域名**：免费泛解析域名，可用于 Let's Encrypt 证书申请

2. **Trojan 配置**：
   - `ssl.cert` 和 `ssl.key` 必须指向有效证书
   - 密码格式为 UUID
   - `local_port: 443` 是默认监听端口
   - TeddySun trojan 镜像**不支持 WebSocket**

3. **nginx 下载服务**：
   - 配置文件必须挂载到 `/etc/nginx/conf.d/default.conf`
   - HTML 文件必须放到 `/usr/share/nginx/html/`
   - 修改配置后需 `nginx -s reload`

4. **Docker 网络**：
   - 两个容器需在同一网络（`trojan-net`）才能通信
   - 外部访问通过 `docker-proxy` 暴露端口

5. **Clash 配置**：
   - 原生 Trojan 协议不需要 `network: ws` 字段
   - `skip-cert-verify: true` 用于跳过证书验证
   - `sni` 必须与 TLS 证书域名匹配

## 2026-04-23 15:36 - Xray-core 部署

### 部署步骤

1. 停止并移除旧 trojan 容器
2. 创建 xray 网络和目录结构
3. 编写 Xray Trojan 入站配置
4. 复制 TLS 证书到 xray 目录
5. 启动 Xray-core 容器（端口 443）
6. 启动 nginx 下载服务（端口 8080）

### Xray-core 配置

`/root/vpn-server/xray/config/config.json`：
```json
{
    "log": {
        "loglevel": "warning"
    },
    "inbounds": [{
        "port": 443,
        "protocol": "trojan",
        "settings": {
            "clients": [{
                "password": "12a90b0b-0465-429f-8f36-14b2ebdaef5d",
                "level": 0
            }]
        },
        "streamSettings": {
            "network": "tcp",
            "tlsSettings": {
                "cert": "/etc/xray/certs/fullchain.pem",
                "key": "/etc/xray/certs/privkey.pem"
            }
        }
    }],
    "outbounds": [{
        "protocol": "freedom"
    }]
}
```

### 注意事项

- Xray Trojan 配置中 `password` 是字符串，原版 trojan 是数组
- Trojan 协议已被 Xray 标记为 deprecated，建议后续迁移到 VLESS
- 客户端连接测试需要用户使用 v2rayN 进行验证

### 背景

原 TeddySun trojan 方案客户端连接持续超时，需要切换到更稳定的 Xray-core + v2rayN 方案。

### 关键发现

#### 1. Xray-core 优势

- Xray-core 是 v2ray-core 的升级版（forked from v2fly-core）
- 原生支持 Trojan、VLESS、VMess 等多种协议
- 支持 XTLS、REALITY 等高级功能
- Docker 官方镜像：`ghcr.io/xtls/xray-core`
- 其他可用镜像：`teddysun/xray`、`v2ray/official`

#### 2. Trojan 协议对比

| 实现 | 来源 | WebSocket 支持 | 特点 |
|------|------|----------------|------|
| trojan-gfw | 原版 Trojan | 否 | 纯 Trojan 协议 |
| Xray-core | v2ray 衍生 | 是 | 支持 fallback |

#### 3. Xray-core Trojan 入站配置

```json
{
  "inbounds": [{
    "port": 443,
    "protocol": "trojan",
    "settings": {
      "clients": [{
        "password": "12a90b0b-0465-429f-8f36-14b2ebdaef5d",
        "level": 0
      }],
      "fallbacks": [{
        "dest": 80
      }]
    }
  }]
}
```

**与原版 trojan 的区别**：
- `password` 是字符串而非数组
- 支持 `fallbacks` 回落分流
- 支持 `level` 用户等级

#### 4. v2rayN 客户端

- Windows/macOS/Linux GUI 客户端
- 支持 Xray-core、sing-box 等内核
- 支持 Trojan、VLESS、VMess 等协议
- GitHub: https://github.com/2dust/v2rayN
- Android 对应: v2rayNG

#### 5. 官方文档

- Xray-core: https://xtls.github.io
- V2Fly 指南: https://guide.v2fly.org/
- v2rayN Wiki: https://github.com/2dust/v2rayN/wiki

### 方案决策

**服务端**：Xray-core (Docker) + Trojan 入站协议
**客户端**：v2rayN + Trojan 协议

**原因**：
- Xray-core 的 Trojan 实现更成熟，支持更多特性
- v2rayN 与 Xray-core 同源，兼容性更好
- 两者都是活跃维护的项目

## v2rayN 客户端测试过程

### 测试环境

- **客户端**：Windows 10/11，v2rayN 7.20.4
- **服务端**：Ubuntu 22.04，Docker + Xray-core
- **网络**：家庭宽带（IP: 59.72.97.73）

### 测试方法

#### 1. 订阅导入测试

1. 在 v2rayN 中使用 `服务器 → 从 URL 导入（订阅）`
2. 订阅地址：`http://74.48.78.17:8080/subscribe`
3. 成功导入 Trojan 节点：`VPN-Trojan(74-48-78-17.nip.io:443)`

#### 2. 速度测试

- **TCLPING**：显示 200-600ms 延迟（TCP 握手成功）
- **真连接延时**：-1ms（表示无法建立连接）

#### 3. 手动 curl 测试

在 Windows CMD 中执行：
```cmd
set http_proxy=
set https_proxy=
curl -vk https://74.48.78.17:443 --connect-timeout 5
```

**结果**：连接超时

### 服务端日志分析

**服务端检查点**：
```bash
# 容器运行状态
docker ps | grep xray

# Xray 日志
docker logs xray

# NAT 规则
iptables -t nat -L DOCKER -n -v
```

**发现**：
- 端口 8080 DNAT 有 18 个包（正常）
- 端口 443 DNAT 只有 1 个包（异常）
- 服务端日志显示从未收到客户端的实际连接

### 问题诊断

1. **TCP 连接可以建立**：TCLPING 显示延迟
2. **TLS 握手失败或协议解析失败**：服务端无对应日志
3. **可能原因**：
   - 客户端代理设置未清除
   - v2rayN 本地端口 7890 被占用
   - 网络环境封锁 VPN 流量

### 调试步骤

#### 步骤 1：清除代理设置
```cmd
set http_proxy=
set https_proxy=
```

#### 步骤 2：检查端口占用
```cmd
netstat -ano | findstr 7890
```

#### 步骤 3：直接测试端口
```powershell
Test-NetConnection -ComputerName 74.48.78.17 -Port 443
```
